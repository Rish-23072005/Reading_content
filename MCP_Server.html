01 — Introduction

What is MCP?
The USB Standard for AI

An AI model is only as powerful as the world it can touch. MCP is the protocol that gives AI hands, eyes, and memory beyond the training data.

Imagine buying a new keyboard. You don't worry about whether it works with your laptop — USB guarantees it will. Now imagine every external tool an AI could ever need — a database, a file system, a calendar, a code runner — having the same guarantee. That's what the Model Context Protocol (MCP) is: a universal, open standard that lets any AI model communicate with any external tool, using a single shared language.

Before MCP, every AI integration was bespoke. You wanted Claude to read your Google Drive? You wrote a custom integration. You wanted it to query your Postgres database? Another custom layer. Every tool meant reinventing the wheel. MCP ended that era.

Anthropic open-sourced MCP in late 2024, and it has since become the de-facto standard for AI tool connectivity. The protocol defines how AI hosts (like Claude) discover, communicate with, and use external servers — whether those servers run locally on your machine or remotely in the cloud.

Core definition

MCP is a client-server protocol built on JSON-RPC 2.0 that defines how AI models discover and invoke external capabilities — tools, data sources, and prompt templates — through a standardized message-passing interface.

02 — Architecture

The Architecture:
Hosts, Clients & Servers

MCP defines three distinct roles in every interaction. Understanding who owns what responsibility is the foundation of building reliable MCP systems.

MCP splits the world into three clear roles, each with precisely defined responsibilities. Confusing these roles is the single most common source of architectural mistakes in MCP implementations.

The Host

The host is the application the end-user interacts with — Claude.ai, a VS Code extension, a custom desktop app. The host is the trust anchor. It decides which MCP servers are allowed to connect, whether sampling requests get approved, and which file system paths are accessible. The host is never optional and never neutral — it is the gatekeeper.

The Client

The client lives inside the host and manages the protocol conversation with a single server. One host can spawn multiple clients — one per server. The client handles connection lifecycle, message serialization, capability negotiation, and routing. End users never interact with it directly.

The Server

The server is an independent process that exposes capabilities — tools, resources, and prompts — to the AI via MCP. Servers are typically small, focused programs. A server might expose only three tools. It doesn't know or care what AI model is using it; it just speaks MCP.

03 — Transport Layer

Transport Layer:
How Messages Actually Travel

MCP is transport-agnostic by design. The same protocol runs over a local process pipe or across the internet — and choosing the right transport matters for performance, security, and deployment.

At its lowest level, MCP needs a way to move bytes between client and server. The protocol defines two official transports, each optimized for a different deployment scenario.

stdio — Standard I/O (Local)

When the MCP server runs on the same machine as the host, stdio is the recommended transport. The client launches the server as a child process. JSON messages flow through stdin (client → server) and stdout (server → client). stderr is reserved for logs. It's brutally simple, requires zero networking, and is inherently sandboxed to the local machine.

HTTP + SSE — Streaming over the Web

For remote servers — hosted APIs, cloud services, shared company tools — MCP uses HTTP with Server-Sent Events (SSE). The client sends requests as HTTP POST calls. The server streams responses back using SSE, which maintains a persistent one-way channel from server to client over a single HTTP connection. This enables progress notifications and streaming results without WebSocket overhead.

Fig 3.1 — stdio vs HTTP+SSE: local and remote transport mechanisms side by side

A third transport, WebSocket, is under discussion in the MCP specification for bidirectional streaming use cases. For now, stdio is the default for local development and HTTP+SSE for production deployments.

04 — Message Format

The Message Format:
JSON-RPC 2.0 Explained

Every single message in MCP — from hello to goodbye — follows the JSON-RPC 2.0 specification. Understanding this format is understanding the heartbeat of the entire protocol.

JSON-RPC 2.0 is a lightweight remote procedure call protocol encoded in JSON. It was chosen for MCP because it's simple, language-agnostic, and widely supported across every programming ecosystem. Every message is a JSON object with a fixed structure.

The Three Message Types

MCP uses exactly three JSON-RPC message shapes. Every message you will ever send or receive in MCP is one of these three:

// 1. REQUEST — expects a response (has id + method + params)
{
  "jsonrpc": "2.0",
  "id": 42,
  "method": "tools/call",
  "params": { "name": "read_file", "arguments": { "path": "/notes.txt" } }
}

// 2. RESPONSE — reply to a request (has id + result OR error)
{
  "jsonrpc": "2.0",
  "id": 42,
  "result": { "content": [{ "type": "text", "text": "Hello world" }] }
}

// 3. NOTIFICATION — fire-and-forget (has method, NO id)
{
  "jsonrpc": "2.0",
  "method": "notifications/cancelled",
  "params": { "requestId": 42 }
}

Fig 4.1 — The three JSON-RPC 2.0 message types: Request (paired), Response (paired), Notification (fire-and-forget)

Critical rule

The id field is the only thing that links a response to its request. Clients maintain an in-flight map of id → promise. When a response arrives, the id resolves the right promise. IDs can be strings or integers — but must be unique within a session.

05 — Core Primitives

Core Primitives:
Tools, Resources & Prompts

MCP servers expose exactly three types of capabilities. Every feature you will ever build in MCP maps to one of these three primitives — and choosing the right one matters enormously.

The three primitives are the vocabulary of MCP. Understanding their differences — and when to use each — is essential for designing clean, composable server APIs.

Tools — Actions

Tools are functions the AI can call to do things. They are the most powerful primitive. A tool might send an email, query a database, execute code, create a file, or call an external API. Every tool has a name, a human-readable description (which the AI reads to decide when to use it), and a JSON Schema describing its inputs.

Resources — Data

Resources are read-only data sources the AI can access. Unlike tools, they are identified by URIs (like file:///home/user/notes.txt or db://customers/id/42). Resources don't execute logic — they return data. The AI can list available resources and read any of them. Resources can also be subscribed to for live change notifications.

Prompts — Templates

Prompts are reusable, parameterized prompt templates stored on the server. A company might store its code review prompt, its customer response template, or its debugging checklist as MCP prompts. The AI retrieves them on demand. Prompts can accept arguments to customize their output.

06 — Initialization

Initialization & Capability Negotiation

Before a single tool can be called, the client and server must shake hands. This initialization algorithm runs exactly once per session and determines everything that's allowed afterward.

The initialization handshake is the foundational algorithm of MCP. It's a 4-step protocol that exchanges version information, advertises capabilities, and establishes what each party can and cannot do for the rest of the session. Every subsequent behavior is governed by its result.

Fig 6.1 — The 4-step initialization handshake: all subsequent behavior is governed by this exchange

After the handshake, both sides store each other's declared capabilities. The rule is absolute: before calling any feature, you must check if the other side declared that capability during initialization. Calling tools on a server that didn't declare capabilities.tools will return error -32601.

07 — Tool Execution

Tool Execution:
The 9-Step Pipeline

A tool call feels instantaneous to the user, but under the hood it passes through nine distinct stages — each with its own logic, error modes, and opportunities to fail gracefully.

Tool execution is the most complex algorithm in MCP. When Claude decides to use a tool, the system orchestrates a precise sequence that spans capability checking, schema validation, transport, deserialization, and AI feedback. Understanding each stage lets you debug failures, optimize performance, and design better tools.

Fig 7.1 — The complete 9-step tool execution pipeline: from capability check to AI feedback loop

The most critical distinction in this pipeline is between step 5 (client-side schema validation) and step 7 (server execution). A validation failure at step 5 never reaches the server — it's surfaced directly to the AI model. A tool failure at step 7 returns isError: true in the result body — the MCP call itself succeeded, and the AI can choose how to respond.

08 — Sampling

Sampling: Servers That
Ask Claude to Think

Sampling inverts the MCP request direction: instead of Claude calling a server, the server calls Claude. This single feature unlocks fully autonomous multi-step AI agents.

In normal MCP, Claude decides when to call a tool. In sampling, an MCP server decides when to invoke Claude's reasoning. The server sends a sampling/createMessage request to the client, which routes it through the host's approval layer before the AI ever sees it.

This enables powerful agentic patterns: a server can fetch data, ask Claude to classify or summarize it, use that reasoning to decide on next steps, call more tools, and loop — all without user intervention at each step.

Fig 8.1 — Sampling flow: server → host gate (approval + model selection) → AI → result back to server

Design principle

Servers must never assume sampling will be approved. The host can reject any sampling/createMessage call. Robust servers design fallback behavior for when reasoning is unavailable — simpler rule-based logic, a cached result, or a graceful failure message to the user.

09 — Reliability

Error Handling, Cancellation
& Progress Reporting

Production-grade MCP systems handle three reliability concerns that tutorials ignore: how errors are classified, how long-running tasks are cancelled, and how progress is surfaced to users.

Error Classification: Two Completely Different Systems

MCP has two error systems that must not be confused. Protocol errors live in the error field of the JSON-RPC response — they mean the MCP call itself failed (bad method name, bad params, server crashed). Tool errors live inside the result field with isError: true — the MCP call succeeded, but the tool's logic failed (file not found, API timeout, invalid input). The handling for each is completely different.

Fig 9.1 — Protocol errors vs tool errors: two completely separate error systems with different handling logic

Cancellation

Any in-flight request can be cancelled by sending a notifications/cancelled notification with the target request's ID. This is fire-and-forget — no acknowledgment is expected. The server should stop processing, but it's best-effort. The client must always handle the race condition where a result arrives after a cancellation was sent — it must silently discard the late result.

Progress Reporting

When calling a tool, include a _progressToken in the _meta field. The server can then send notifications/progress messages with a progress number (0–1) and optional total, allowing the UI to display a progress bar. Servers are not required to honor progress tokens, but well-designed servers should.

10 — Implementation

Building Your First
MCP Server: End-to-End

Theory solidifies into understanding when you build. Here is a complete mental model and code skeleton for your first MCP server — a file reader with three tools.

A minimal MCP server needs to handle five things: initialize, list tools, call tools, handle errors, and shut down cleanly. Here's the architecture before a single line of code:

Fig 10.1 — Anatomy of a minimal MCP server: router → handlers → tool implementations → error wrapping

// Minimal MCP server skeleton (TypeScript / Node.js)

import { Server } from "@modelcontextprotocol/sdk/server/index.js";
import { StdioServerTransport } from "@modelcontextprotocol/sdk/server/stdio.js";

const server = new Server(
  { name: "file-reader", version: "1.0.0" },
  { capabilities: { tools: {} } }
);

// 1. Declare your tools
server.setRequestHandler(ListToolsRequestSchema, async () => ({
  tools: [{
    name: "read_file",
    description: "Read a text file from disk",
    inputSchema: {
      type: "object",
      properties: { path: { type: "string", description: "Absolute path" } },
      required: ["path"]
    }
  }]
}));

// 2. Implement your tools
server.setRequestHandler(CallToolRequestSchema, async (req) => {
  const { name, arguments: args } = req.params;
  if (name === "read_file") {
    try {
      const text = await fs.readFile(args.path, "utf-8");
      return { content: [{ type: "text", text }] };
    } catch (e) {
      // Tool error — not a protocol error
      return { content: [{ type: "text", text: e.message }], isError: true };
    }
  }
  throw new McpError(ErrorCode.MethodNotFound, `Unknown: ${name}`);
});

// 3. Connect and run
await server.connect(new StdioServerTransport());

With this skeleton, Claude can now discover your server's tools, call them, receive results, and handle errors — all over the standardized MCP protocol. From here, you extend by adding more tools, declaring resources, subscribing to file system events, or enabling sampling for AI-assisted processing pipelines.

Next steps
